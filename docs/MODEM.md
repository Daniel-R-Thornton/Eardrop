# Modem Specification

## Overview

The Eardrop modem currently has two active physical layers that share the same file-transfer protocol stack:

| Layer | Modulation | Current role |
|-------|-----------|--------------|
| BPSK | 4-tone BPSK with a continuous pilot | The more mature acoustic path used for live file transfer when OFDM is off |
| OFDM/QPSK | Native-rate, multi-tone QPSK | The higher-throughput path used when the UI enables OFDM |

The pilot tone remains the phase/amplitude reference for both modes. In BPSK mode the tones are relative to the pilot; in OFDM mode the tones are absolute frequencies on a 50 Hz grid in the 2–4 kHz band.

There are two protocol stacks in the codebase:

| Stack | Primary use | Notes |
|-------|-------------|-------|
| Encoder/Decoder | Self-test and loopback tests | Older reference path with its own framing and ECC |
| TxEngine/RxEngine | Production transfer path | Used by the UI and worker; handles both BPSK and OFDM |

## Configuration

The main modem config lives in `src/modem/types.ts` and is built for the UI by `src/ui/controllers/buildModemConfig.ts`.

| Field | Current default | Notes |
|-------|-----------------|-------|
| `sampleRate` | 3200 for BPSK defaults | OFDM uses the hardware sample rate from the browser |
| `pilotFreqHz` | 600 for BPSK | OFDM uses the OFDM default pilot of 1900 Hz |
| `pilotAmplitude` | 0.4 for BPSK | OFDM uses `OFDM_DEFAULTS.pilotAmplitude = 2.0` |
| `toneCount` | 4 for BPSK | OFDM defaults to 32 tones and accepts 8/16/32 |
| `diversityMode` | false | Repeats each frame three times when enabled |
| `symbolsPerSec` | 25 | Largely a BPSK-era knob; OFDM uses its own timing constants |

## BPSK mode

BPSK uses a pilot tone plus data tones at offsets from the pilot. The default offsets are still the legacy 100 Hz spacing values:

```text
TONE_OFFSETS = [100, 200, 300, 400]
```

That yields a 4-tone layout such as 700/800/900/1000 Hz when the pilot is at 600 Hz. The implementation uses the shared `PhaseAcc` oscillator in `src/modem/dsp/oscillator.ts`, so the phase contract remains consistent across modulation and demodulation.

## OFDM/QPSK mode

OFDM is implemented as a native-rate path in the production TxEngine/RxEngine pipeline. The relevant defaults are:

```text
OFDM_SYMBOL_MS = 20
OFDM_CP_MS = 5
OFDM_DEFAULTS.pilotFreqHz = 1900
OFDM_DEFAULTS.pilotAmplitude = 2.0
OFDM_DEFAULTS.toneCount = 32
OFDM_TUNING.syncBurstSymbols = 24
OFDM_TUNING.trainingSymbols = 12
OFDM_TUNING.syncMinFrames = 8
OFDM_TUNING.tailSilenceSymbols = 6
```

The OFDM tone grid is generated by `ofdmToneFrequencies()` and uses an absolute 50 Hz spacing in the 2–4 kHz band, with the pilot placed below the data-tone block.

## Frame format

The production path uses atomic frames with a sentinel scanner. The current OFDM frame format is:

```text
[SENTINEL 3B][BCH_HEADER 24B][RS_PAYLOAD 208B] = 235 bytes
```

This carries a 160-byte payload per frame with four RS(52,40) blocks. The BPSK path still uses the same sentinel-based frame pipeline, but the frame size and ECC balance differ from the OFDM path.

## Receive behavior

The receiver behavior is split by mode:

- BPSK: wake on energy, detect the warble preamble, perform calibration, then enter a frame scan.
- OFDM: detect a sync burst via tone energy and cyclic-prefix correlation, train per-tone equalizers from the burst, then decode data frames.

The OFDM path uses a sync-once-then-coast model rather than continuous per-symbol timing tracking. That is a deliberate tradeoff in the current implementation.

## Current status

- BPSK is the more mature acoustic transfer path and is still the default when OFDM is off.
- OFDM is implemented end to end and covered by in-memory tests, including sync, channel drift, pilot-level, throughput, and frame-geometry cases.
- Live acoustic verification is still the next step for the OFDM path, as documented in `STATE.md`.

This provides 3× redundancy at the cost of 3× transmission time. No voting/consensus — just repetition with first-copy-wins.

## Pilot Handling

**Encoder/Decoder path**: Pilot frequency is set from `ModemConfig.pilotFreqHz` (600 Hz default). A second-order PLL (`PilotPLL` in `modem/pilot.ts`) tracks pilot phase for amplitude estimation, but the data path uses raw I/Q directly — no pilot-relative rotation is needed because the phase contract ensures consistent signs.

**TxEngine/RxEngine path**: Same pilot frequency. PLL is initialized on first sample and updated continuously for amplitude tracking.

**OFDM path**: Pilot at 1900 Hz (below the data band). Used for phase-drift correction per symbol. Pilot amplitude is 2.0 (fixed in OFDM_TUNING, separate from the BPSK pilot amplitude).

| Path | Scheme | Correction Power |
|------|--------|-----------------|
| Encoder/Decoder | BCH(31,16) + interleave depth 8 | 3 bit errors per 31-bit codeword |
| TxEngine/RxEngine | BCH(63,30) × 3 + RS(52,40) × 4 | 3 bit errors per BCH codeword + 6 byte errors per RS block |

## OFDM/QPSK Mode (Native-Rate)

Time-domain OFDM with QPSK per subcarrier. Symbols are defined in milliseconds rather than samples, so the protocol adapts automatically to any hardware sample rate (48000 Hz, 44100 Hz, etc.).

### Architecture

- **Symbol length**: 20 ms + 5 ms cyclic prefix = 25 ms total (~40 sym/s)
- **Modulation**: Direct cosine synthesis at exact tone frequencies (no IFFT)
- **Demodulation**: Goertzel / toneIQ bank at exact tone frequencies (no FFT)
- **Tone grid**: Multiples of 50 Hz (1000 / 20 ms), guaranteeing orthogonality at any sample rate
- **Pilot**: Continuous pilot at 1900 Hz, used for per-tone channel equalization and phase drift correction

### Protocol Constants

| Parameter | Value |
|-----------|-------|
| `OFDM_SYMBOL_MS` | 20 ms |
| `OFDM_CP_MS` | 5 ms |
| `pilotFreqHz` | 1900 Hz |
| `pilotAmplitude` | 2.0 |
| `toneSpacingHz` | 50 Hz |
| `toneStartHz` | 2000 Hz |
| `toneCount` default | 32 |
| Sync burst | 24 symbols (~600 ms) |
| Training symbols | 12 |
| Sync detection windows | 8 consecutive above-threshold |
| Raw bitrate (16 tones) | 1280 bps |
| Raw bitrate (32 tones) | 2560 bps |
| Net payload rate (32 tones) | ~1707 bps (166 B/s) |
| Frame format V2 | [SENTINEL 3B][BCH 24B][RS(52,40)×4 = 208B] = 235B carrying 160 payload B |
| Payload density | 68% |
| Symbol alignment | CP correlation (single-shot, no rolling tracking) |
| Channel tracking | Decision-directed per-tone equalization (α = 0.003) |
| Phase drift correction | Pilot-referenced, frequency-proportional |

### Tone Grid

Tones are at `2000 + n * 50` Hz for n = 0 … toneCount-1. With default 32 tones: 2000, 2050, 2100, …, 3550 Hz. The pilot is at 1900 Hz, below the data band. All frequencies — pilot and tones — are exact multiples of 50 Hz for orthogonality with the 20 ms symbol.

### Throughput Benchmark (2000-byte file, 48 kHz)

| Config | 16 tones | 32 tones |
|--------|----------|----------|
| Baseline (45ms symbol, 79B frame) | 41.5 B/s | 80.8 B/s |
| + 4 RS blocks (235B frame) | 48.6 B/s | 92.6 B/s |
| + 20ms symbol (25ms total) | **87.4 B/s** | **166.7 B/s** |
| Overall gain vs baseline | ×2.1 | ×2.1 |

### Sample Rate Adaptivity

Symbol length in samples is `Math.round(sampleRate * OFDM_SYMBOL_MS / 1000)` and CP in samples is `Math.round(sampleRate * OFDM_CP_MS / 1000)`. At 48000 Hz: 960-sample symbol + 240-sample CP = 1200 samples. At 44100 Hz: 882 + 221 = 1103 samples. The Goertzel/toneIQ bank operates at the exact tone frequencies regardless of sample rate.

### Timing Recovery: Sync-Once-Then-Coast

The OFDM receiver currently lacks a rolling timing tracking loop. After initial acquisition:

1. **CP correlation** determines the symbol boundary at sync time
2. All subsequent windows are consumed at a fixed stride (`sps` samples)
3. **Pilot phase drift correction** handles sub-carrier rotation but does **not** adjust the window stride
4. **Decision-directed channel tracking** adjusts per-tone phase/amplitude estimates symbol-by-symbol
5. A **sync-loss watchdog** resets to WAITING if no frames arrive within ~15 seconds

The 5 ms CP provides a 240-sample timing guard at 48 kHz. At 50 ppm clock drift between devices (~2.4 samples/s), the system has ~100 seconds before the window drifts outside the CP — sufficient for typical file transfers. Future work could add a timing error detector + FLL for indefinite transmission.
